# Build Discovery Milestone 1 Assumptions

Generated for review on 2026-07-09. This is an assumption inventory, not an
acceptance claim.

## Scope

- Milestone 1 is level 200 PvM Iop only.
- The query contract carries `level`, but only `level=200` is accepted.
- Only one element is accepted at a time.
- Supported elements are Strength, Intelligence, Chance, and Agility.
- Non-Iop classes, non-PvM mode, non-200 levels, and multi-element queries are
  intentionally rejected.
- Current confidence is highest for steered melee Strength Iop scoring. Other
  element results are generated by profile substitution and still need benchmark
  review.

## AP / MP / Range

- AP, MP, and Range are minimum targets with hard caps, not exact targets.
- For level 200 / level 100+ characters, Milestone 1 valid query bounds are
  AP 7-12, MP 3-6, Range 0-6.
- Characters below level 100 normally start at 6 AP, but non-200 generation is
  outside Milestone 1.
- Surplus AP/MP/Range is valid if it does not exceed the cap, but should only
  receive small marginal score.
- Defaults remain 11 AP, 6 MP, 0 Range for level 200 Iop.

## Budget / Availability

- Budget means accessibility, not exact kama price.
- Availability v0 is hardcoded and intentionally coarse.
- Tier 1 includes mounts, trophies, and normal equipment not assigned to higher
  tiers.
- Tier 2 adds pets, petsmounts, and accessible Dofuses.
- Tier 3 adds exos and prysmaradites.
- Tier 4 adds opti assumptions such as Ochre, Vulbis, legendary/special-effect
  items, and other advanced items.
- Budget tiers below 3 force effective `exoPolicy=none` even when the query asks
  for `allow` or `opti`.

## Dofus / Trophy / Exos

- Accessible Dofuses currently include Crimson, Emerald, Turquoise, Ice, and
  Dolmanax.
- Ochre and Vulbis are treated as opti Dofuses.
- Shaker, Nomad, Jackanapes, and Voyager are preserved as AP/MP enabler
  candidates.
- Dofus/trophy exclusivity rules are not yet modeled by the prototype.
- Exos are generated only as missing +1 AP, +1 MP, or +1 Range.
- At most one AP exo, one MP exo, and one Range exo are applied.
- Exos are only applied to eligible equipment that does not already grant that
  stat.

## Scoring

- Damage, survivability, and utility are hardcoded prototype scores.
- Strength has the most manually steered scoring; other elements mostly swap
  primary stat and elemental damage profile.
- `damageSurvivabilityPreset` is validated and stored, but current prototype
  behavior is still driven mainly by explicit score weights.
- Conditional special effects are modeled only where explicitly coded.
- Cloudy, Vulbis, Ochre, and AP prysmaradite effects use simplified expected
  value assumptions.

## Search / Packages

- Search is set/package-aware and bounded, not exhaustive.
- Set packages currently focus on 2-piece and 3-piece packages.
- Strong cross-set paths should be seeds/shortcuts, not exclusive replacements
  for partial-set, single-set, locked-item, and fallback search.
- Normal gear candidate horizons include the level 180-200 bucket for level 200
  and selected evergreen/action-stat exceptions.
- Dofus/trophy/prysmaradite combinations are bounded and should preserve core
  staples, AP/MP enablers, special-effect Dofuses, and benchmark-required
  combinations even when cheap pre-score ranks them low.

## Current Hard Limits

- Prototype defaults: `limit=5`, `top_k=25`, `beam_width=250`,
  `per_signature_cap=40`, `relevant_set_limit=60`, `max_shared_items=10`.
- Backend API caps: `limit<=5`, `topK<=100`, `beamWidth<=1000`,
  `perSignatureCap<=200`, `relevantSetLimit<=200`.
- Search/package hard limits include global package limit 500, triple source
  limit 250, triple scan limit 3,000,000, direct completion seed limit 700,
  Dofus pool limit 22, Dofus combo limit 500, and direct final score limit 1000.

## Review Flags

- Prove every supported element/budget/AP/MP/Range family with benchmarks or
  diagnostic coverage before calling Milestone 1 done.
- Add production-derived benchmarks only after local correctness coverage is
  broad enough to interpret misses.
- Keep this document current when assumptions move from hardcoded prototype
  behavior into reviewed product behavior.
